## Architecture Overview

This is a **standalone installable Pi package** (pattern: [pi-suite](https://github.com/dvictor357/pi-suite)), not a manual `.pi/` drop-in. Pi's package manager reads the root `package.json` → `pi.extensions` array and auto-discovers extension directories. Installed via `pi install git:github.com/<user>/agent-summoner` or `pi install .` for local dev.

```
agent-summoner/
├── package.json           # pi.extensions: ["./extensions"]
├── tsconfig.json          # single typecheck gate over all .ts
├── AGENT.MD               # agent identity document
├── README.md              # install + usage docs
├── extensions/
│   └── orchestrator/
│       ├── index.ts       # Entry: exports default function(pi), wires everything
│       ├── ledger.ts      # Ledger data structure, read/write, persistence (pi.appendEntry)
│       ├── agents.ts      # Agent registration: Scout, Crafter, Gatekeeper + user-defined
│       ├── planner.ts     # Topological sort (Kahn), phase building, plan presentation
│       ├── commands.ts    # /summon, /watch, /back commands
│       └── ui.ts          # Status widget (ctx.ui.setWidget), watch-mode (ctx.ui.custom)
├── docs/                  # prd.md, flow.md, plan.md, todo.md, guide.md
└── self/                  # atlas.md, engineering.md, docs-policy.md
```

**Key difference from original plan:** No `.pi/extensions/` manual install. The repo IS the package. Pi discovers the extension via `package.json` → `pi.extensions` → `extensions/orchestrator/index.ts`. No `core/` directory needed (single extension — per pi-suite architecture doc: "core owns only what crosses an extension boundary").

## Key Technical Decisions

| Decision | Choice | Reasoning |
|---|---|---|
| Package pattern | Standalone installable (pi-suite style) | `pi install git:…` / `pi install .`. No manual `.pi/` drop-in. |
| Install discovery | `package.json` → `pi.extensions: ["./extensions"]` | Pi auto-discovers `index.ts` in listed dirs. Same mechanism as pi-suite. |
| Ledger persistence | `pi.appendEntry("ledger-update", …)` (Option B) | Survives `/reload`, crash recovery. Data stored in session JSONL, not LLM context. |
| Ledger replay algorithm | Iterate `sessionManager.getEntries()` in tree order, latest-wins per file path | Deterministic replay. Uses id/parentId tree ordering (already present in session format). |
| Scout cache invalidation | Proactive — Crafter calls `markDirty(path)` on write; also dirties reverse-dependency files | Prevents stale replanning. One `Set<string>` in shared module state. Transitive: if A imports B and B changes, A's cached graph is also invalidated. |
| Gatekeeper baseline | Run before Phase 1 execution | Enables informed "was this already broken?" decisions. |
| Crafter crash recovery | Timeout detection → mark files `pending` → re-assign | Prevents deadlocked phases. Default timeout: 5 min, configurable via `AGENT_TIMEOUT_MS`. |
| Testing framework | `node:test` + `node:assert` via `tsx` (zero deps) | Built into Node 20+. Matches pi-suite convention. No jest/vitest dependency. |
| File mutation safety | `withFileMutationQueue` on every Crafter write | Guide §5 — non-negotiable safety net. Serializes writes to same file even across parallel tool calls. |
| Trust-mode parameter type | `StringEnum` not `Type.Union`/`Type.Literal` | Guide §10 gotcha — `Type.Union` breaks on Google's API. `StringEnum(["trust", "checkpoint"])`.

## Trade-offs

**Optimizing for:** correctness and safety over raw speed.
**Sacrificing:** some execution speed (baseline test run, proactive cache invalidation, richer report mechanism).

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Scout's AST parser fails on non-standard JS/TS | Graceful fallback: regex-based results with `confidence: "low"`. Main Agent treats low-confidence as single-phase. |
| `npm install` fails in Phase 0 | Abort task, report error to user. No partial state. |
| Two Crafters touch same file despite phase model | `withFileMutationQueue` serializes writes. Richer report conflict check catches before proceeding. |
| LLM doesn't emit parallel tool calls | System works serially. Slower but correct. Model behavior, not architecture bug. |

## Phase 1 — Scout

### Task 1.1: File/symbol search
- Accept `scope` (directory glob) and optional `pattern` (symbol/regex to find)
- Use `node:fs` glob + line-by-line grep (zero deps — no external glob library needed for basic `**/*.ts`)
- Return matched files with line numbers, not full contents
- Apply `truncateHead` from `@earendil-works/pi-coding-agent` (max 200 lines / 10KB before truncation)
- [ ] Symbol search returns correct file+line for known codebase
- [ ] Output truncated when exceeding 200 lines or 10KB
- [ ] Empty result when nothing found (empty array, not error)
- [ ] Scope limits search to requested directory

### Task 1.2: AST-level import/export parsing
- Parse JS/TS for `import`/`export` statements using regex-first approach, then `@babel/parser` for precise graphs
- **Prefer built-in or lightweight**: try regex for ESM `import … from`, `export …`; fall back to `@babel/parser` for ambiguous cases
- Build dependency graph: `{ [file]: { exports: string[], importedBy: string[] } }`
- Export `markDirty(path)` — also dirties all files in `importedBy` list (transitive invalidation)
- Graceful fallback: parse errors → regex extraction with `confidence: "low"`
- [ ] Correct graph for known multi-file codebase (A→B→C, diamond pattern)
- [ ] Unparseable files get regex fallback with `confidence: "low"`
- [ ] Circular dependencies detected and flagged (don't infinite-loop)
- [ ] `markDirty` dirties reverse dependencies

### Task 1.3: Session-scoped cache
- `Map<string, { graph, mtime, confidence }>` — keyed by file path
- On request: check cache → check `dirtyScoutCache` → re-parse if dirty or missing
- Export `markDirty(path)` from shared state (used by Crafter after writes)
- Export `invalidateScope(scope)` for full rescans
- [ ] Cache hit avoids re-parse (verify via timing or call count spy)
- [ ] Dirty file (in `dirtyScoutCache`) re-parsed
- [ ] `dirtyScoutCache` cleared per-file after successful re-parse
- [ ] Reverse-dependency files also invalidated when dependency changes

### Task 1.4: Scout tool registration + test
- Register `summon_scout`: `{ scope: string, pattern?: string }`
- Return: `{ graph, slices, confidence }`
- [ ] Tool appears and returns correct graph
- [ ] Cache works on repeated calls
- [ ] Unit tests: graph building, cache hit/miss, dirty invalidation

## Phase 4 — Gatekeeper

### Task 4.1: Pre-task baseline test run
- Run full test suite before Phase 1
- Store: `{ file, passing }[]`
- [ ] Baseline captured before edits
- [ ] Failures don't block execution

### Task 4.2: Full test suite execution (post-task)
- Run after all phases complete: `npm test` (or project's test command, detected from package.json)
- Compare each failure against baseline
- Classify each failure into one of three buckets:
  - **in-scope**: file was `pending`/`in_progress`/`done` in Ledger (touched by this task)
  - **out-of-scope-new**: file NOT in Ledger AND was passing in baseline → we broke it
  - **out-of-scope-pre-existing**: file NOT in Ledger AND was failing in baseline → already broken
- [ ] All three categories correctly classified
- [ ] Baseline comparison correctly detects new vs pre-existing
- [ ] Empty baseline (no prior tests) → all failures treated as out-of-scope-new

### Task 4.3: In-scope failure handling
- 🙈: auto-fix + re-test, max 3 attempts
- 🔍: present to user, ask before fixing
- [ ] Trust mode auto-fixes silently
- [ ] Checkpoint mode pauses and asks
- [ ] Max 3 retries enforced
- [ ] Fix attempts logged

### Task 4.4: Out-of-scope failure handling

## Development Workflow

### Local development (no install needed)

```bash
# Load extension directly for dev (hot reload with /reload)
pi -e ./extensions/orchestrator/index.ts

# Or install from local path (persists across sessions)
pi install .
pi install . -l    # project-local only
```

### Package scripts (in package.json)

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "test": "node --import tsx --test extensions/orchestrator/*.test.ts",
    "test:watch": "node --import tsx --test --watch extensions/orchestrator/*.test.ts"
  }
}
```

### Testing strategy details

| Layer | Framework | Location | When |
|---|---|---|---|
| Unit (pure logic) | `node:test` + `node:assert` | `extensions/orchestrator/*.test.ts` | As each module is built (Phase 0-7) |
| Integration (multi-module) | `node:test` with mocks for Pi APIs | `extensions/orchestrator/*.test.ts` | Phase 8 |
| End-to-end (real Pi session) | Manual test scenarios | N/A | Phase 8 (acceptance) |

**Mock strategy:** Pi APIs (`pi.appendEntry`, `ctx.sessionManager`, `ctx.ui`, etc.) are mocked in unit tests. Create a `test-helpers.ts` with mock factories. Integration tests in Phase 8 wire real modules together with mocked Pi boundary.

**Test file co-location:** Unit tests live in same directory as source (`extensions/orchestrator/`), named `<module>.test.ts`. This is the pi-suite convention.

### Install for users

```bash
# From GitHub
pi install git:github.com/<user>/agent-summoner

# From local path
pi install ./agent-summoner

# Uninstall
pi remove agent-summoner
```
